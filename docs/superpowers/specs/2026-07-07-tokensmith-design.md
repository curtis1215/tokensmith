- **日期**：2026-07-07
- **狀態**：設計已確認，待寫實作計畫
- **類型**：終端機（TUI）經營模擬遊戲
- **一句話**：一款跑在終端機裡的 AI 公司經營模擬遊戲；背景常駐 daemon 會採集你**真實**使用 Claude Code / Codex 消耗的 token，轉換成遊戲內的研發燃料，驅動你的虛擬 AI 公司成長。

---

## 1. 概述

玩家經營一間 AI 公司。核心是兩條資源迴圈的交織：

- **燃料迴圈（真實使用）**：你在 IDE 用 AI coding 工具寫 code → 背景 daemon 偵測 token 消耗 → 注入研發燃料。
- **複利迴圈（遊戲內經營）**：訓練模型 → 獲取用戶 → 訂閱收入 → 再投資算力/團隊 → 更強的模型。

遊戲以「陪跑」形式常駐背景，隨你真實 coding 默默成長；當你坐下來打開 TUI，進入完整經營介面做決策。

v1 聚焦兩條業務線：**訓練 AI 模型** 與 **算力/服務器經營**。晶片設計留待 v2。

## 3. 系統架構

### 3.1 進程拓撲

單一 Go binary、多子命令，邏輯上三個進程角色：

```
tokensmith daemon           ← 無頭背景進程（launchd 自啟），狀態的唯一真相來源
tokensmith  (tokensmith tui) ← Bubble Tea TUI，attach 到 daemon 看即時畫面、下指令
tokensmith daemon install    ← 安裝/移除 launchd agent
```

- TUI 啟動時若偵測不到 daemon，會**自動 spawn 一個 detached daemon**，所以不「安裝」也能玩；想要開機自啟＋長期陪跑才 `install`。
- TUI 是**薄客戶端**：只讀狀態、送指令，永遠不直接改狀態。所有變更都走 daemon → sim。
- daemon 為單例（single-instance），以 lock file / socket 存在偵測避免重複啟動。

### 3.2 內部層（package）

| 層 | 套件 | 職責 |
|---|---|---|
| **採集** | `internal/ingest` | `TokenSource` 介面 + `ClaudeCodeAdapter` / `CodexAdapter`。fsnotify 監看 JSONL，解析出 `TokenEvent`。每檔一個 cursor（byte offset + inode）持久化，重啟不重複計、離線只讀新增段。 |
| **模擬** | `internal/sim` | **純函式、確定性**的遊戲核心。`Tick(state, dt, events) → (newState, domainEvents)`。兩迴圈：token→R&D（燃料）、經營→現金+算力（複利）。離線結算 = 快轉 tick。 |
| **存檔** | `internal/store` | SQLite（純 Go `modernc.org/sqlite`，免 cgo、好跨平台編譯）。存 GameState 快照、ingest cursors、事件日誌。 |
| **daemon 執行期** | `internal/daemon` | 串起 ingest→sim→store，跑 tick loop，開 IPC server（Unix socket），發桌面通知（抽象成 `Notifier` 介面，macOS 用 osascript/terminal-notifier）。 |
| **TUI** | `internal/tui` | Bubble Tea + Lipgloss。連 daemon IPC、訂閱狀態流、畫儀表板、送指令。 |
| **共用** | `internal/model` | 跨層共用的資料型別（`GameState`、`TokenEvent`、IPC 訊息、command）。 |
| **平衡** | `internal/balance` | 集中所有數值常數與曲線（成本、成長率、轉換率、tick 比例）。 |

### 3.3 資料流

```
Claude Code / Codex logs
      │ fsnotify + tail
      ▼
 Ingest adapters ──TokenEvent──► Sim engine ──► Store (SQLite)
                                    │  ▲
                        domain event│  │ command (IPC)
                                    ▼  │
                             Notifications   TUI client
```

### 3.4 關鍵設計性質

1. **Sim 純且確定性** → 離線結算就是「重放」；不依賴 daemon/TUI 就能單元測試。
2. **Daemon 是唯一真相** → TUI 只是讀/命令客戶端，可多開。
3. **Ingest 冪等** → cursor 防重複計；重啟安全。
4. **Adapter 可插拔** → 未來加 Cursor/Copilot 不動 sim。

## 4. 採集層（Ingest）

### 4.1 `TokenSource` 介面

```go
type TokenEvent struct {
    Source       string    // "claude-code" | "codex"
    Timestamp    time.Time // 來自 log 的事件時間
    InputTokens  int
    OutputTokens int
    // 其餘欄位（cache tokens、model 名）保留擴充
}

type TokenSource interface {
    Name() string
    // Watch 啟動監看，透過 channel 送出正規化事件；ctx 取消時停止。
    Watch(ctx context.Context, from Cursor, out chan<- TokenEvent) error
    // Backfill 讀取自某 cursor 以來的歷史事件（離線結算用）。
    Backfill(from Cursor) ([]TokenEvent, Cursor, error)
}
```

### 4.2 v1 兩個 adapter

- **`ClaudeCodeAdapter`**：監看 `~/.claude/projects/**/*.jsonl`。每行是一筆 JSON；解析 assistant 訊息行的 `message.usage.{input_tokens, output_tokens, cache_creation_input_tokens, cache_read_input_tokens}`。
- **`CodexAdapter`**：監看 `~/.codex/sessions/**/*.jsonl`（rollout 檔）。解析其 token usage 事件。

> 兩者實際 JSON 結構需在實作階段以真實樣本確認；adapter 各自負責容錯（格式不符的行略過、不中斷）。

### 4.3 冪等 / cursor

- 每個被監看檔案維護一個 `Cursor{ Inode, ByteOffset }`，持久化於 store。
- 只讀 offset 之後的新增 bytes；事件以（檔案 + offset）為鍵，重讀安全。
- 檔案輪替 / inode 改變時重置該檔 cursor。
- 這讓「離線只讀新增段」與「重啟不重複計」用同一套機制達成。

## 5. 模擬引擎（Sim）

### 5.1 GameState（概念）

```
GameState {
    Time:        累計遊戲時間 / 目前遊戲日期
    Resources:   { RnD, Cash }
    Compute:     { 訓練容量, 推理容量, 訓練負載, 推理負載 }
    Research:    { Manpower, 研究員清單, 效率加成 }
    TechTree:    各維度已解鎖加成 / 等級
    Models:      []Model{ Gen, 四維品質{能力,成本,安全,速度}, 主打區隔, 上線時間, 是否下架 }
    Market:      { TAM, 各區隔前沿水準, 各區隔用戶分佈 }
    Competitors: []Competitor{ 名稱, 性格archetype, 研發曲線, 目前各維度水準, 各區隔市佔 }
    Infra:       { 租用{訓練,推理}, 機房[]{電力容量,空間容量}, 服務器[]{晶片型號,數量,所屬池}, 電費率 }
    Staff:       { 各職能人數與素質, 明星員工[]{技能,薪資,留任度} }
    Milestones:  已達成集合
    Prestige:    { Patents, 已購永久升級 }
    Cursors:     每個 token 來源的讀取進度
}
```

### 5.2 Tick 函式

- 簽章：`Tick(state, dt, events) → (newState, domainEvents)`，**純函式**。
- 每 tick 依序：
  1. 吃 `events`（token）→ 依轉換公式加 R&D。
  2. 研發人力 → 產出 R&D（`研發人力 × 效率 × dt`）。
  3. 推進進行中的訓練工作（消耗**訓練算力** + 時間），完成則模型上線。
  4. 推進對手研發曲線 → 更新對手模型 / 各區隔市場前沿水準；對手可能對玩家明星員工出價挖角。
  5. 依各模型相對品質 → 更新各區隔市佔與用戶數（含前沿上升造成的相對老化流失）。
  6. 依用戶數計算推理負載；**推理算力**容量 < 負載 → 服務品質降、額外流失。
  7. 依訂閱（用戶數 × 玩家定價）+ API 用量 → 產生現金收入。
  8. 扣除持續成本（租算力費、電費、薪資）。
  9. 出租閒置算力 → 被動現金。
  10. 檢查里程碑達成 → 產 domainEvent。
- `domainEvents`（訓練完成、里程碑、破產警告…）由 daemon 轉成通知。

### 5.3 時間模型 / 離線結算

- 即使沒有 token，遊戲照跑 real-time tick（模型閒置收入、研發人力產出照進）。
- game-time 比例可調（config，例如 1 real sec ≈ 1 game 分鐘）。
- **離線結算**：daemon 啟動時
  1. 計算離線實際時長，取 `min(離線時長, 上限)`。
  2. `Backfill` 各 token 來源的歷史事件。
  3. 從上次存檔狀態快轉 tick 到現在（含 token backlog）。
  - 因 Tick 是純函式且確定性，快轉 N tick 的結果須等同逐 tick 跑 N 次（property test 保證）。

## 6. 遊戲經濟（v1）

### 6.1 三種資源

| 資源 | 來源 | 用途 |
|---|---|---|
| **R&D 研發點數** | 研發人力（主幹）+ 真實 token（燃料） | 訓練模型、解鎖科技、升級 |
| **現金 Cash** | 經營（訂閱收入、出租算力） | 買算力/晶片、蓋服務器、招人、行銷 |
| **算力 Compute** | 租來 or 自建服務器 | 訓練吃算力、推理也吃算力 |

### 6.2 R&D 產生（兩條進料）

| 進料 | 機制 | 意義 |
|---|---|---|
| **研發人力（主幹）** | `R&D/tick = 研發人力 × 效率係數`。研發人力 = 研究員數 × 素質 × 基建加成。 | 有養研發團隊才有穩定產出；沒團隊時幾近為零。 |
| **真實 token（燃料）** | 實際 coding 的 token → 注入爆發性 R&D（v1 **相加**）。 | 真實使用加速/灌注研發。 |

**節奏設計**：早期尚無研發團隊，R&D 幾乎全靠你真實 coding 的 token（真實使用直接幫你起步）；中後期養起團隊後有穩定涓流，真實 token 轉為「衝刺加成」。

> 待定小項：token 是「相加一筆 R&D」還是「短時間放大研發人力產出（相乘 buff，更貼近『燃料』語感）」。v1 先用**相加**，日後改 buff 容易。

### 6.3 核心迴圈

```
真實 token ─► R&D ─┐
研發人力 ──────────┤─► 訓練模型（吃 R&D + 佔用算力一段時間）
自建/租 Compute ───┘        │
                           ▼
                     模型上線 → 用戶成長(S曲線) → 訂閱收入 Cash/tick
                           │                      │
                     推理續吃算力            Cash → 算力/招人/行銷/服務器
```

### 6.4 產出端細化：模型 & 用戶市場

#### 6.4.1 品質維度（4 維）

每個模型有四個品質維度：

- **能力 Capability** — 前沿智能水準
- **成本效率 Efficiency** — 每次推理的成本 / CP 值
- **安全性 Safety** — 可靠、可控、企業合規
- **速度 Speed** — 延遲 / 吞吐

#### 6.4.2 市場區隔（3 區隔）

用戶市場分三個區隔，各自對維度的權重不同（權重為 balance config，可調）：

| 區隔 | 看重維度 | 收入型態 |
|---|---|---|
| 消費者 Consumer | 能力 + 速度 | 訂閱月費 |
| 企業 Enterprise | 安全 + 穩定 | 訂閱月費（單價高） |
| 開發者 Developer | 成本 + 速度 | API 用量計費 |

#### 6.4.3 訓練決策（疊乘公式）

訓練一個模型時，玩家：

1. 選**世代 / tier**（決定各維度的總量天花板）。
2. 把 **R&D + 算力預算分配**到四個維度（決定維度分佈 / 定位）。

模型某維度最終值 ≈ `該維度預算佔比 × 科技樹該維度加成 × 世代上限係數`。三者疊乘：

- **預算分配** = 每次訓練的主動定位（塑造分佈）
- **科技樹加成** = 長期投資、跨所有模型永久生效（拉高效率/上限）
- **世代 / tier** = 總量天花板

科技樹投資越深，同樣預算能訓出越強的模型。**R&D 成本與訓練時間解耦**：R&D 決定「能不能訓、訓多好」；訓練時間 = `訓練工作量 W / 投入訓練GPU數`（投更多算力就更快完成）。高世代需前置里程碑 / 科技解鎖。

#### 6.4.4 競爭市場與市佔

- **完全競爭**：一個會隨時間成長的總用戶池（TAM），數家**具名對手公司**共享。
- **市佔分配**：在每個區隔內，各公司「最佳對應模型的加權品質分」決定該區隔用戶的分配比例（相對品質 → 相對市佔）。
- **市佔排行榜**：可看各公司在各區隔 / 總體的市佔。

#### 6.4.5 性格化對手（腳本化成長）

- 數家真實 AI 公司對手，各有**性格 archetype** 與**研發曲線**：
  - 例：`OpenAI`（激進衝能力/前沿）、`DeepSeek`（低價衝成本效率）、`Anthropic`（安全，主打企業）。完整 7 家陣容見 §17.1。
- 依固定曲線定期推出新模型、推升自己的維度水準；**不針對玩家 rubber-band**（公平可預測、好平衡）。
- 對手推新 → 該區隔前沿上升 → 搶走用戶。

#### 6.4.6 前沿上升與模型老化（相對衰退）

- 模型**絕對品質固定**於訓練當下。
- 整體「市場前沿水準」隨對手推進而上升；模型相對前沿的落後幅度越大 → 各區隔市佔越低 → **用戶漸流失**（相對老化，非絕對衰減）。
- 因此必須**持續推新世代模型**汰換 / 補位 → 這是 token 燃料長期有用的根本原因。

#### 6.4.7 多模型組合（產品線佈局）

- 可**同時營運多個模型**，各主打不同區隔（例：消費旗艦 Gen4 重能力 + 開發 API Gen3 重成本）。
- 舊模型可**留著吃殘餘用戶**（收入遞減）或**下架**以省推理算力。
- 策略：用有限的 R&D / 算力，在哪些區隔佈幾條產品線、何時汰換。

#### 6.4.8 收入模型（雙軌 + 玩家訂價 + 推理吃算力）

- **玩家訂價**：玩家為每個模型自訂價格。各區隔有「參考價」；`需求乘數 = (參考價 / 你的定價)^彈性`。
  - 定價 > 參考 → 新用戶變慢、流失略升、單客收入高（高毛利路線）。
  - 定價 < 參考 → 新用戶變快、衝量（薄利多銷，DeepSeek 打法）。
  - 新用戶速率 ∝ 品質吸引力 × 需求乘數 × 行銷加成。
- **訂閱軌**（消費者 / 企業）：Cash/tick = 該區隔用戶數 × 你的定價。
- **用量軌**（開發者 / API）：Cash 隨推理量 × 你的 API 計費。
- **推理負載 ↔ 算力**：所有活躍用戶產生推理負載、消耗**推理算力**。
  - 容量 ≥ 負載 → 正常服務；容量 < 負載 → 服務品質下降（延遲 / 降級）→ 額外流失。
- 這把**產出端（用戶）與投入端（算力）咬合**：擴用戶必須同步擴算力。

### 6.5 投入端細化：算力 · 服務器 · 機房

#### 6.5.1 兩種算力池

算力分成兩個獨立的池，都可由「租」或「自建」提供：

- **訓練算力**：被訓練中的模型消耗。
- **推理算力**：被服務中的活躍用戶消耗（推理負載）。

規劃兩種容量是核心決策：訓練算力不足 → 訓不動新模型；推理算力不足 → 服務品質降 → 用戶流失。

#### 6.5.2 租 vs 自建

- **租**：無 capex、按 tick 扣現金、隨時增減、可分別租訓練 / 推理容量。早期首選、彈性高、單價貴。
- **自建**：三層鏈（晶片 → 服務器 → 機房），一次性 capex + 持續電費，單價低很多（規模經濟），但要規劃容量。
- 「租的彈性 vs 自建的規模經濟」是投入端主要張力。

#### 6.5.3 三層鏈：晶片 → 服務器 → 機房

- **晶片（config 驅動）**：屬性 = 世代 / 訓練型 or 推理型 / 算力 / 功耗 / 佔用空間 / 價格。
  - 新世代隨里程碑（或時間）解鎖：更高算力/瓦、更貴；舊世代降價 → 呼應硬體進步，與市場前沿上升平行。
- **服務器**：一台裝 N 顆同型晶片（**基礎密度 N=8，硬體基建科技可提升到 12 / 16 / 24**）。capex = 晶片成本 + 機箱；貢獻算力到對應池（訓練 / 推理）；佔用機架空間。密度越高 → 同樣空間塞更多算力。
- **電費**：`電費/day = Σ(每片 GPU 功耗 kW) × 電價`——隨你裝的 GPU 數量增加（散熱已併入電價，不獨立成約束）。
- **機房**：提供兩個容量上限——**電力容量（kW，不可超過）** 與 **機架空間（格）**。
  - 容量滿了 → 擴建現有機房或蓋新機房（capex）加容量。
- **出租閒置算力**：任一池容量沒用滿時可出租換被動現金。

### 6.6 團隊：職能員工 + 明星個體

#### 6.6.1 四種聚合職能

用「招 N 個 T 級員工」聚合管理，各職能貢獻對應加成：

| 職能 | 貢獻 |
|---|---|
| 研究員 Researcher | → 研發人力（被動 R&D 產出） |
| 工程師 Engineer | → 基建效率（訓練加速、算力效率、電費↓） |
| 營運 Ops | → 服務穩定（推理效率、降低服務不足流失） |
| 行銷 Marketing | → 用戶獲取（加速市佔轉換、行銷加成） |

研發人力 ≈ `Σ(研究員 × 素質) × 基建加成 + 明星研究員加成`。

#### 6.6.2 具名明星員工（人才爭奪戰）

- 除聚合員工外，有**可挖角的明星個體**（例：首席科學家、明星系統工程師），各有獨特技能與**大幅 targeted 加成**（如首席科學家 → 大幅拉高能力維度與研發人力）。
- **取得**：從人才池簽約（高簽約金 + 高薪/tick）；挖角對手在職明星成本更高。
- **流失風險**：對手也會對你的明星出價挖角；留任取決於薪資競爭力 / 士氣。低薪 → 有被挖走風險。
- 把團隊系統與競爭層咬合：明星人才在你與對手之間流動。

#### 6.6.3 薪資（持續成本）

- 每名員工（聚合 + 明星）有 salary/tick，計入每 tick 的持續成本。
- 過度擴編 → 薪資壓垮現金流；裁員可降成本但損加成。

## 7. 進度 · 科技樹 · Prestige · 平衡

### 7.0 估值（valuation）

驅動里程碑與 prestige 的核心指標。估值 ≈ 收入水準、用戶總數、資產（自建算力 / 機房）、市佔的加權組合（balance config）。

### 7.1 科技樹（四分支主題樹）

用 **R&D** 沿分支解鎖永久研究；可偏科。R&D 同時是「訓練模型」與「投資科技樹」的花費 → 產生「這局多訓一個模型 vs 永久變強」的張力。

| 分支 | 內容 |
|---|---|
| 演算法 Algorithms | 四維品質的產出效率加成（能力 / 成本 / 安全 / 速度各自）、解鎖更高模型世代 |
| 硬體基建 Infra | 算力效率↑、電費↓、機房效率、**服務器密度↑（8→12→16→24）**、解鎖更高晶片世代（銜接 v2 晶片設計） |
| 商業營運 Business | 用戶獲取↑、付費意願 / 參考價↑、API 計費↑、降低流失 |
| 對齊安全 Alignment | 安全維度效率↑、企業信任↑、抗事故（降低負面事件衝擊） |

- 節點成本隨深度指數上升；偏科能更快在某定位取得優勢。
- 科技樹加成**跨所有模型永久生效**（呼應 6.4.3 疊乘公式）。

### 7.2 里程碑

達成 → 解鎖新內容 + 一次性獎勵（現金 / R&D / 科技點）。三條里程碑軸：

- **估值**：$1M → $10M → $100M → $1B → $10B → $100B → $1T
- **模型世代**：Gen1 → Gen2 → … → GenN（解鎖更高上限；高世代需前置科技 / 里程碑）
- **關鍵成就**：首個付費用戶、首座自建機房、首位明星科學家、市佔登頂某區隔、**AGI 達成**（終極里程碑）

里程碑也是解鎖 prestige、新晶片世代、新科技分支節點的關卡。

### 7.3 Prestige（永久升級樹）

- **解鎖條件**：估值首次突破門檻（例如 $1B）才能 prestige。
- **重開清除**：R&D、現金、算力、機房、已訓練模型、聚合員工全部歸零。
- **換取**：依本局峰值估值換永久貨幣 **專利 Patents**（次線性）：
  `專利 = floor( sqrt( 本局峰值估值 / K ) )`
- **永久升級樹**（patents 投入、跨局保留、以數值加成為主）：
  - 起始資源（起始現金 / R&D）
  - 全域倍率（R&D× / 現金× / 訓練速度× / 算力效率×）
  - 明星員工留任加成、起始已解鎖科技
- **驅動力**：帶加成重跑，前期遠快 → 衝更高估值 → 換更多專利。無盡重玩循環。

一句話：**prestige = 用「這局爬多高」兌換「下局變多強」，然後歸零重爬。**

### 7.4 節奏 arc（陪跑週級）

目標：**一局 ≈ 數天～一兩週真實時間**，公司在你日常 coding 期間穩定成長。

| 階段 | 樣貌 |
|---|---|
| 早期（第 1～2 天） | 無研發團隊，R&D 幾乎全靠真實 token；租算力；訓 Gen1～2 模型主打單一區隔搶第一批用戶。 |
| 中期（數天） | 養起研發團隊（穩定 R&D 涓流）；自建首座機房；多產品線佈局；科技樹鋪開；招募第一位明星；衝 $100M～$1B。 |
| 晚期（～兩週） | 規模化機房、明星團隊、四分支科技深化；市佔登頂、衝 AGI；首次 prestige 進入下一循環。 |

### 7.5 平衡曲線哲學

- 成本 / 門檻**指數上升**、產出**次線性**，形成經典 idle 的「持續但遞減」推進。
- token→R&D 有每日軟帽 / 遞減，避免爆量破壞曲線。
- 所有數值集中在 `internal/balance`（見 §12），先以週級節奏標定，再迭代微調。

## 8. 玩家體驗：token 手感 · 產業事件 · onboarding

### 8.1 token 手感（真實 coding 被看見）

coding 時 daemon 偵測到 token 流，遊戲主題化呈現成「研發衝刺」：

- **即時能量流**：R&D 即時跳動，顯示是哪個工具在灌注（`⚡ Claude Code +1.2k R&D/s`、`⚡ Codex …`）。
- **連續 streak**：連續 coding 天數給 R&D 加成（`🔥 連續 5 天 ×1.3`），鼓勵日常使用；中斷則 streak 重置 / 衰減。
- **重大突破**：單次 coding 累積達門檻 → 觸發「突破」桌面通知 + 對某科技分支的短期加速（`💥 突破！演算法分支加速`）。
- 這是 Tokensmith 的招牌手感：真實使用被強烈、即時地看見與回饋。

### 8.2 防爆表 / 防作弊（輕量，為平衡而非防弊）

- 單機個人遊戲，作弊只損自己 → 不做侵入式反作弊。
- **每日軟帽 / 遞減曲線**（balance）：一天灌注的 R&D 邊際遞減，避免某天狂 coding 直接爆破曲線。目的是**平衡**，不是抓弊。
- token 事件冪等（§4.3 cursor）確保重讀不重複計。

### 8.3 線上 vs 離線感受

- **線上（開著 TUI）**：即時看 R&D 能量流、用戶 / 市佔跳動、事件即時彈出。
- **離線（關著）**：下次啟動 daemon 回溯結算，TUI 進場先給「**歡迎回來**」摘要——離線期間賺了多少現金、研發人力產出多少 R&D、處理了多少 token backlog、發生哪些事件、對手推進多少。經典 idle「回來領獎」的滿足感。

### 8.4 首次開場 onboarding

- 輕度引導、非長教學：新玩家開局有一個明確首要目標——**訓練第一個模型、搶到第一個付費用戶**。
- 隨進度用 tooltip / 提示逐步揭露系統（先模型與租算力，再團隊、自建、科技樹、prestige），避免一次資訊爆炸。
- 早期 R&D 幾乎全靠真實 token → 自然把玩家導向「去 coding 幫公司起步」，第一次體驗招牌機制。

### 8.5 產業事件（有玩法影響 + 選擇）

定期發生 AI 產業事件，對系統有實質影響，部分帶玩家選擇：

- **供應鏈**：晶片短缺 → 自建成本↑ / 晶片降價潮。
- **對手動態**：對手醜聞 / 安全事故 → 其用戶流向市場（`[接收企業客 / 觀望]`）；對手大突破 → 搶你用戶。
- **研究突破**：突破論文 → 某科技分支成本↓ / 加速。
- **安全事故**：你的模型出事 → 信任 / 用戶受損；**對齊安全**科技分支（抗事故）可降低衝擊 → 讓該分支有實質價值。
- **市場 / 法規**：市場榮枯、法規變動 → TAM / 成本波動。

事件用 domain event → 桌面通知 / TUI 事件列；帶選擇者暫存待玩家決策。

## 9. 存檔（Store）

- **SQLite 單檔**（純 Go `modernc.org/sqlite`，免 cgo），WAL 模式；僅 daemon 存取（單寫者）。
- 位置：macOS `~/Library/Application Support/tokensmith/tokensmith.db`（跨平台以 XDG / OS 慣例決定）。
- 只有 daemon 讀寫；TUI 一律透過 IPC 取狀態（§10）。

### 9.1 Schema

```sql
-- 跨局永久（prestige 不清）
CREATE TABLE meta_progression (
  id            INTEGER PRIMARY KEY CHECK (id = 1),  -- single row
  patents       INTEGER NOT NULL DEFAULT 0,
  prestige_tree TEXT    NOT NULL DEFAULT '{}',       -- JSON: nodeId → level
  lifetime      TEXT    NOT NULL DEFAULT '{}',       -- JSON: 累計統計、歷史最佳估值
  updated_at    INTEGER NOT NULL
);

-- 目前這一局的完整狀態快照（prestige 時重置）
CREATE TABLE game_state (
  id         INTEGER PRIMARY KEY CHECK (id = 1),     -- single row, 最新快照
  schema_ver INTEGER NOT NULL,
  snapshot   TEXT    NOT NULL,                        -- JSON: 整個 GameState（不含 cursors / meta）
  game_time  REAL    NOT NULL,                        -- 遊戲時鐘
  saved_at   INTEGER NOT NULL                         -- Unix 時間，離線結算基準
);

-- 採集游標（冪等；ingest 高頻更新，與快照解耦）
CREATE TABLE cursors (
  source      TEXT    NOT NULL,                       -- "claude-code" | "codex"
  file_path   TEXT    NOT NULL,
  inode       INTEGER NOT NULL,
  byte_offset INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL,
  PRIMARY KEY (source, file_path)
);

-- 事件日誌（append-only；「歡迎回來」摘要 + 稽核）
CREATE TABLE event_log (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  game_time REAL    NOT NULL,
  wall_ts   INTEGER NOT NULL,
  type      TEXT    NOT NULL,                          -- TrainDone / Milestone / IndustryEvent...
  payload   TEXT    NOT NULL                           -- JSON
);

-- 版本 / 遷移
CREATE TABLE meta (
  key   TEXT PRIMARY KEY,                              -- save_version / game_version / created_at
  value TEXT NOT NULL
);
```

### 9.2 設計要點

- **快照為 JSON blob**：GameState 是內聚結構、sim 為唯一真相 → 存成單筆 JSON 最省、schema 穩定；GameState 形狀演進靠 `schema_ver` + 遷移處理，不需逐表正規化 models / servers / staff。
- **cursors 獨立**：ingest 每 tick 更新游標而不必重寫整份快照；重啟 / 離線結算只讀游標之後的新增段（§4.3 冪等）。
- **meta_progression 跨局保留**：prestige 時 `game_state` 重置、`meta_progression`（專利 / 永久樹 / 生涯統計）保留。
- **原子性**：快照定期寫（每 N 秒）+ 關鍵事件即時落地，皆在 transaction 內；崩潰後從最後快照 + 游標 backfill 復原。
- **離線結算**：啟動讀 `saved_at` 算離線時長 → backfill token 事件 → 從快照快轉（§5.3）。

## 10. Daemon 執行期

- 串接 `ingest → sim → store`，跑固定 tick loop。
- `Notifier` 介面發桌面通知（訓練完成、里程碑、破產警告）。macOS 實作用 osascript / terminal-notifier。
- 單例保護：lock file / socket 偵測。

### 10.1 IPC 協定

- **傳輸**：newline-delimited JSON（JSONL）over Unix domain socket（`~/Library/Application Support/tokensmith/daemon.sock`）。
- **信封**：每則訊息含 `type`；請求含 `id` 供回應對應。支援多客戶端（狀態廣播）。
- **一致性**：所有變更只由 daemon 內 sim 執行；TUI 純讀 State + 送 Command，永不本地改狀態（§3.4）。

**Client → Daemon**

| 訊息 | 說明 |
|---|---|
| `Hello{clientVersion}` | 握手 |
| `Subscribe{}` | 開始接收狀態流 |
| `Command{id, cmd}` | 下指令（cmd 見下表，皆經 sim 驗證） |
| `Resync{}` | 要求重送完整快照 |

指令 `cmd`：

| cmd | 參數 |
|---|---|
| TrainModel | gen, name, alloc{能力,成本,安全,速度}, segment |
| SetPrice | modelId, price |
| RetireModel | modelId |
| RentCompute | pool(training / inference), delta |
| BuyServer | chipType, count, datacenterId |
| ExpandDC / BuildDC | datacenterId?, power?, slots? |
| HireStaff / FireStaff | role, tier, count |
| SignStar / PoachStar | starId |
| AdjustSalary | targetId, salary |
| UnlockTech | nodeId |
| ResolveEvent | eventId, choice |
| Prestige / BuyPrestigeNode | nodeId? |

**Daemon → Client**

| 訊息 | 說明 |
|---|---|
| `Welcome{gameVersion, saveVersion}` | 回應 Hello |
| `State{seq, snapshot \| delta}` | 訂閱後先送完整快照，之後送 delta（節流 ~4–10 Hz） |
| `Ack{id}` / `Error{id, code, message}` | 指令結果 |
| `Event{event}` | domain event：TrainDone / MilestoneReached / EventTriggered(需選擇) / PoachAlert / BankruptcyWarning |

- **錯誤碼**：InsufficientCash / InsufficientRnD / CapacityExceeded / LockedByMilestone / InvalidCommand…（驅動 TUI 的 ⚠ 提示）。

### 10.2 v1 實作：輕量採集 daemon（單進程 TUI 變體）

v1 的 TUI 採單進程、直接驅動 sim（見 §11 實作）。因此 daemon 不擁有 sim、不跑 socket，改採**「daemon 唯一採集、TUI 唯一消費」**的解耦設計：

- **`tokensmithd`**（獨立 binary）：背景常駐，每 ~5s poll `~/.claude` + `~/.codex`，持久化 cursor（含 inode，重啟不重讀/不漏），把 token 累積量原子寫進 ledger；單例保護用 PID/lock file。
- **ledger**（`~/Library/Application Support/tokensmith/ledger.json`）：`{cumInputTokens, cumOutputTokens, updatedAt}`，單調遞增。
- **TUI sidecar meta**（`meta.json`，與 save 同目錄）：`{consumedIn, consumedOut, lastRealUnix}`；sim 維持純函式，牆鐘時間只存在此。
- **消費**：TUI 每 tick 取 `delta = ledger.cum − consumed` 當 token events 餵進 `sim.Tick`，再把 consumed 推進到 ledger.cum——線上/離線統一。
- **離線結算**（TUI 開啟時）：`offlineDelta = ledger.cum − consumed`、`elapsed = now − lastRealUnix`；分段 Tick（每段 ≤1h，首段帶 offlineDelta tokens）推進世界（訓練完成、用戶成長、對手推進、離線 R&D 入帳），顯示總結。
- **降級**：ledger 過期（>30s 未更新）或缺失 → TUI fallback 回內建 poller，遊戲仍可獨立玩。
- **一致性**：daemon 只寫 ledger、TUI 只寫 save/meta（不同檔），無鎖、無 race。
- **v1 不含**：桌面通知、daemon 自動啟動、多客戶端（皆需 §10.1 全 IPC 版，留 v2）。

## 11. TUI

### 11.1 技術與原則

- Bubble Tea（Elm 式狀態機）+ Lipgloss（樣式）+ Bubbles（表格 / spinner / 進度條）+ Harmonica（動畫）。
- 連上 daemon IPC，訂閱狀態流即時渲染；下指令一律送 daemon，不本地改狀態（薄客戶端）。
- **頂部常駐資源條**：現金 / R&D 速率 / 兩池算力利用率 / 估值 / 日期 / streak——所有分頁可見。
- **一致操作語言**：`Tab` 切頁、`↑↓` 選項、`Enter` 確認、括號內單鍵快捷、`Esc` 取消。
- **壓力提示**：⚠ 標紅即時提示需處理處（推理接近上限、模型衰退、明星低薪、對手將發表）。
- coding 時 R&D 速率條即時跳動、事件列冒出「💥 突破（來自哪個工具）」——招牌手感的呈現點。

### 11.2 分頁結構（6 頁）

`總覽 · 模型 · 市場 · 算力 · 團隊 · 科技`（事件用彈窗 + 總覽事件列；里程碑 / prestige 掛總覽與科技頁）

| 頁 | 內容 |
|---|---|
| 總覽 | 公司 KPI（估值 / 用戶 / 排名 / 營收）、進行中訓練、兩池算力壓力、主戰場市佔、下個里程碑、事件列（含待決策 `[選項]`） |
| 模型 | 營運中模型清單（世代 / 四維 / 區隔 / 用戶 / 營收 / 相對前沿衰退）+ 訓練新模型的「預算分配」介面 |

| 團隊 | 四職能聚合（人數 / 素質 / 加成）、明星員工（留任 / 挖角）、招募 / 加薪 / 解雇 |

## 14. v1 範圍界定

### In scope（v1）

- Go 單一 binary：`daemon` / `tui` / `daemon install` 子命令。
- 採集層：Claude Code + Codex 兩個 adapter，被動讀 log、冪等 cursor。
- Sim：訓練模型 + 算力/服務器兩條線的完整迴圈。
- R&D 雙進料（研發人力 + token 燃料）。
- 產出端：4 維品質 × 3 區隔 × 性格化具名對手、市佔競爭、相對老化、多模型組合。
- 雙軌收入（訂閱 + API 用量）、推理負載吃算力。
- 投入端：訓練 / 推理兩算力池、晶片 → 服務器 → 機房三層自建、租 vs 自建、電力 / 空間容量。
- 團隊：研究員 / 工程師 / 營運 / 行銷四職能 + 具名明星員工（可被對手挖角）。
- 科技樹四分支（演算法 / 硬體基建 / 商業營運 / 對齊安全）、里程碑、prestige 永久升級樹（專利）。
- 節奏：陪跑週級（一局 ≈ 數天～一兩週）。
- token 手感：即時能量流、streak、突破通知；離線「歡迎回來」摘要。
- 產業事件（有玩法影響 + 選擇）、輕度 onboarding。
- SQLite 存檔、離線結算、桌面通知。
- Bubble Tea 儀表板 TUI。

### Out of scope（v1，留 v2+）

- **晶片設計** 業務線（設計 → 代工廠 → 自用 / 販售給雲廠商）。詳見下方「v2 預覽」。
- 其他採集來源（Cursor / Copilot…）——介面預留，adapter 不做。
- 跨平台通知（v1 先 macOS）。
- 多人 / 線上排行榜 / 雲端同步（v1 的市佔排行是對 AI 對手，單機）。

### v2 預覽：晶片設計線

第三條業務線，不做進 v1；此處僅勾勒藍圖並確認 v1 已留好接點。聚焦**資料中心晶片的垂直整合**（不含終端 / 邊緣晶片市場）。

**三階段流程**

1. **設計 Design**：投入 R&D + 工程，選架構、製程節點、特化方向（訓練型 / 推理型）。權衡 算力 ↔ 功耗 ↔ 良率 ↔ 設計成本，產出帶自訂（算力 / 功耗 / 空間）屬性的晶片藍圖。
2. **代工 Foundry**：藍圖交付代工廠（流片費 + 每片製造成本 + lead time；良率 yield 影響產出與單價）。製程越先進 → 效能越好但流片越貴、良率越難。
3. **變現 Monetize**：
   - **自用（垂直整合）**：自製晶片組進服務器 → 比市場買的更便宜 / 更強 / 更特化 → 降算力成本、拉高利潤。
   - **外賣**：賣給雲服務廠商 / 市場 → 新收入線（對手也可能採購你的晶片）。

**v1 已留好的接點**

| v1 元件 | v2 延伸 |
|---|---|
| 晶片屬性系統（§6.5.3） | 自製晶片變成此系統新條目，屬性由你決定 |
| 硬體基建科技樹分支（§7.1） | 「解鎖更高晶片世代」延伸為「解鎖自製晶片設計 / 更先進製程」 |
| 兩算力池 + 特化（§6.5.1） | v1 留白的「特化」→ v2 可設計極度偏訓練 / 偏推理的晶片 |
| 對手系統 | 對手可自製或向你採購晶片，競爭延伸到硬體供應鏈 |

**為何留 v2**：v1 已是完整可玩迴圈；晶片設計是規模化後的垂直整合進階玩法，適合首次 prestige 後解鎖或 DLC 式擴充。

### 17.6 Onboarding 逐步腳本

**開場白**：「你剛創立一間 AI 新創。第一個目標：訓練你的第一個模型、搶到第一個付費用戶。」

| # | 步驟 | 引導 / 提示 | 揭露 |
|---|---|---|---|
| 1 | 租算力 | 高亮「算力」頁 → 引導租 1 顆訓練 GPU；tooltip 解釋訓練吃訓練算力 | 開場只顯示 `總覽 · 模型 · 算力` 三頁 |
| 2 | 訓練第一個模型 | 開訓練介面 → 引導預算分配（解說四維 + 區隔）→ 訓一個 Gen1 主打消費者 | tooltip 解說疊乘公式（先簡化） |
| 3 | 去 coding（招牌時刻） | 「訓練要時間——去 IDE 用 Claude Code / Codex 寫 code，看研發能量灌進來」→ 回遊戲看 R&D 即時跳動 + streak 起步 | 第一次體驗 token 招牌手感 |
| 4 | 模型上線 → 首批用戶 | 訓練完成桌面通知 → 模型上線 → 用戶涓流進來 → 首筆營收 | 解鎖「市場」頁 |
| 5 | 定價 | 引導設定價格，解說「定價↑毛利高但成長慢 / 定價↓衝量」 | tooltip 解說需求彈性 |

| 8+ | 自由經營 | 系統隨進度漸次揭露：自建機房、明星挖角、prestige 各由條件 gate | 完整 6 頁全開 |

**原則：** 漸進揭露（先 3 頁、隨里程碑解鎖）；招牌機制前置（第 3 步就推玩家去真實 coding）；早期靠 token 起步自然強化代入；每步一個明確目標、用高亮 + tooltip，不做教學牆。
