# Tokensmith 戰略戰役擴充 — Master Design

**Date:** 2026-07-10
**Status:** Approved in brainstorming; pending written-spec review
**Scope:** A 戰略戰役骨架 → B 動態事件鏈 → C 深度營運
**Implementation planning boundary:** 本文件描述整體終局架構；下一份 implementation plan 僅涵蓋 Phase A。Phase B、C 必須各自另寫 implementation plan。

---

## 1. 背景與問題

Tokensmith 已具備完整的經營模擬骨架：真實 Claude Code／Codex token 轉換為 R&D、模型訓練與發佈、三市場區隔、定價、訓練／推理算力、租用／自建、團隊、科技、具名對手、產業事件、里程碑與 Prestige。

目前限制遊玩時長的主因不是資源或頁面太少，而是中後期的**決策密度與局間差異不足**：

- 里程碑目前主要是估值計數，沒有構成階段任務或勝利路線。
- 對手主要向玩家 frontier rubber-band，缺少能預判與反制的公司戰略。
- 科技、團隊、明星、事件與 Prestige 多為倍率或一次性選項，尚未形成公司 build。
- 模型發佈後缺少成熟、內耗、衍生、承諾與下架的產品生命週期。
- 玩家可能只需等待資源、重複升級，缺少「這次登入必須做什麼決定」的壓力。

本設計不以提高成本或延長倒數硬拖時數，而是把現有系統串成一場有身份、有宿敵、有高潮、有結算的 AI 公司戰役。

## 2. 已確認的產品方向

| 面向 | 決策 |
|---|---|
| 主要優化區間 | 一局中後期的 3–7 個真實日 |
| 登入節奏 | 每天 2–3 次，每次 5–10 分鐘 |
| 核心樂趣 | AI 公司策略戰 |
| 流派約束 | 半鎖定：可營運調整，可付代價正式轉型 |
| 對手 | 有固定性格與公開 Roadmap，可預判、可反制 |
| 終局 | 消費者、企業、開發者三條勝利路線 |
| 擴充範圍 | A 戰略戰役、B 事件鏈、C 深度營運全部納入 |
| 交付順序 | A → B → C；每階段必須獨立可玩、可發布 |

### 2.1 核心玩家承諾

每次登入後，玩家應在 5–10 分鐘內完成：

1. 看懂離線期間發生了什麼。
2. 看見主要宿敵下一步要做什麼。
3. 辨認產品、合約、算力或財務上的最大壓力。
4. 做 2–3 個值得思考的營運決策。
5. 選擇至多一個會影響未來數個週期的高層指令。
6. 離開時清楚知道下一次登入要驗證什麼結果。

### 2.2 非目標

- 不把遊戲改成需要長時間盯盤的即時策略遊戲。
- 不靠增加數十種貨幣或逐項收菜延長時數。
- 不逐名員工排班、逐台伺服器維護或管理大量小合約。
- 不讓對手讀取隱藏玩家選擇後即時生成完美反制。
- 不要求等 ABC 全部完成後才發布第一個有價值版本。
- 不在本輪加入多人、雲端排行榜或 PvP。

## 3. 一局的三幕結構

### 3.1 Day 1：定位

- 玩家完成第一個模型並理解市場後，確立主要公司戰略。
- 顯示本局主要宿敵、攪局者與兩者的初始 Roadmap。
- 完成路線「立足」任務，建立第一個對應市場據點。

### 3.2 Day 2–4：角力

- 推進模型世代、科技、人才與算力選擇。
- 對手依 Roadmap 發表模型、調價、開源、搶人才或擴張。
- 玩家在強化自身路線與花費高層指令反制對手之間取捨。
- 中期可選一個副戰略；必要時可付出代價正式轉型。

### 3.3 Day 5–7：決勝

- 玩家完成路線「擴張」任務後，主要宿敵啟動公開的終局反攻。
- 玩家必須同時達成路線核心條件，並在宿敵攻勢下守住兩個董事會週期。
- 勝利後可 Prestige、繼續無盡經營，或稍後再決定。
- 第 18 個董事會週期後仍無法勝利者，可用較低報酬策略退出，避免壞局無限拖延。

## 4. 董事會週期與雙層時間模型

### 4.1 即時經濟 Tick

保留現有 `sim.Tick()` 的職責：

- token、研究員與明星產生 R&D。
- 訓練工作推進。
- 用戶、營收、租金、薪資、電費與推理負載更新。
- 現有短期 modifier 生效與到期。

高頻 Tick 不直接推進公司戰略、宿敵 Roadmap 或多階段事件鏈。

### 4.2 董事會 Cycle

- 每 **8 小時真實時間**產生一個董事會週期，一天約三個。
- 每 Cycle 玩家可發出至多一項高層指令；一般調價、租算力、招人等營運操作不消耗高層指令。
- Cycle 推進順序固定：
  1. 完成產品、商業與基建管線。
  2. 執行已預告的對手行動。
  3. 推進事件鏈階段。
  4. 更新品牌、企業信任與開發者採用。
  5. 更新路線任務與勝利條件。
  6. 產生不可變的董事會報告。

### 4.3 離線規則

- 啟動時以 wall-clock 計算應補算的 Cycle，最多補算三個。
- 經濟離線結算維持現有上限與規則，不因 Campaign cap 而停止。
- 已承諾的專案與已預告的對手行動可離線完成。
- 離線不得自動接受合約、啟動採購、訓練、轉型、Prestige，或選擇新增支出的事件選項。
- 同一批離線 catch-up 中新產生的重大決策不自動逾時；回到 TUI 後先交給玩家處理。
- 已在先前登入呈現、且明確標示期限的重大決策，逾時只能走免費、保守、不可新增支出的預設結果。
- 同時最多保留一個重大 Pending Decision；有 pending 時不再生成第二個需玩家處理的重大事件，但不阻塞一般經濟與已承諾專案。

## 5. A — 戰略戰役骨架

### 5.1 公司主要戰略

第一個模型上線後才要求選擇主要戰略，讓玩家先看懂四維品質、定價與市場差異。

#### 消費者霸主：前沿品牌

- **擅長：**旗艦發表、品牌熱度、快速用戶成長、大規模推理。
- **義務：**必須持續推新；老化、過載與安全事故對品牌影響較大。
- **核心取捨：**搶先發表或延後換品質；高價品牌或大眾滲透；流量爆發前是否預留推理算力。

#### 企業信任：可信平台

- **擅長：**安全合規、高價長約、穩定現金流。
- **義務：**安全、容量、SLA 與合規門檻；失約造成賠償與信任損失。
- **核心取捨：**接受大單或先補冗餘；為合規犧牲能力或冒險搶時程。

#### 開發者生態：開放平台

- **擅長：**低成本 API、效率、工具鏈與生態採用。
- **義務：**價格天花板低；需維持相容性、效率與穩定吞吐。
- **核心取捨：**開源或閉源；降價搶量或保毛利支撐算力；相容性或快速世代更替。

#### 5.1.1 路線成長

- 每條主要戰略有「立足 → 擴張 → 決勝」三階段。
- 立足與擴張階段各提供一次二選一能力，形成同路線內的不同 build。
- 能力應改變決策條件或風險，不應全部只是 `+10%`：例如發表爆發、品牌衰減、SLA 緩衝、相容性承諾、平價需求增幅等。
- 中期完成擴張任務後可選一個副戰略，只取得其一階能力，不改變主要勝利路線。

階段 gate 的 Phase A 初始值：

- **立足（全路線）：**至少一個該區隔模型上線，且該區隔市佔達 10%；完成後選第一個路線能力。
- **消費者擴張：**消費者市佔達 25%，且推理利用率低於 90% 維持一個 Cycle。
- **企業擴張：**安全品質達 `max(15, 主要宿敵安全品質 × 0.9)`、企業市佔達 20%，且定價不低於有效參考價，維持一個 Cycle。
- **開發者擴張：**效率＋速度加權品質進入前三、開發者市佔達 20%，且定價不高於有效參考價，維持一個 Cycle。
- 完成擴張後選第二個路線能力並開放副戰略；決勝 gate 使用 §5.4 的完整勝利條件。

#### 5.1.2 正常調整與正式轉型

下列行為屬正常營運，不視為轉型：換模型區隔、調價、調整科技優先級、租／買算力、換人才投入。

正式更換主要戰略時：

- 每局最多一次。
- 啟動一個董事會週期的「品牌重定位」專案。
- 現金成本為 `max($20,000, 目前月營收)`，另扣當前 R&D 的 10%；任一資源不足即不可啟動。
- 原主要路線進度降低 30%。
- 決勝階段不可轉型。
- 已取得的舊路線能力在轉型完成後失效；副戰略重新選擇。

#### 5.1.3 路線進度不是貨幣

品牌熱度、企業信任與開發者採用是不可花費的進度 meter。Phase A 以當前狀態與最近兩份 Board Report 重新計算下一個 gate 的完成百分比，因此市佔、品質、定價或服務惡化時會下降，但不形成三種新的囤積資源。Phase B／C 可加入持續數個 Cycle 的事件或承諾 modifier，仍不得直接花費 meter。

### 5.2 對手 Roadmap

#### 5.2.1 每局陣容

- **主要宿敵 ×1：**與玩家主要戰略直接競爭。
- **攪局者 ×1：**從側面改變價格、供應、開源或人才環境。
- **背景公司 ×5：**維持市場與事件存在，但不要求玩家同時追蹤七條完整 Roadmap。

主要宿敵依路線優先配對：

- 消費者：OpenAI、xAI、Gemini。
- 企業：Anthropic、Zhipu、Gemini。
- 開發者：DeepSeek、Qwen、Anthropic。

攪局者從未被選為主要宿敵的不同 archetype 中抽取。

#### 5.2.2 情報層級

玩家總能看見兩步情報：

- **已確認：**下一個 Cycle 的具體行動、目標市場、預估影響與倒數。
- **傳聞：**再下一步的方向與可能影響；準確度可受科技、事件或 Legacy 提升。

每個情報項目必須列出至少兩種可行反制，不代表玩家必須反制。

#### 5.2.3 公平規則

對手可以：

- 依 seed、公司 archetype 與自己的公開狀態推進 Roadmap。
- 根據已結算的公開市場條件選擇預先定義的分支。
- 在玩家領先時啟動提前一 Cycle 公開的追趕專案。

對手不可以：

- 讀取玩家尚未提交的命令或 dialog 選擇。
- 沒有預告便跨世代、瞬間調整所有品質或生成完美克制。
- 使用無上限 rubber-band 把所有維度永久鎖在玩家上方。

### 5.3 高層指令

每 Cycle 最多一項，代表 CEO 層級的資源集中：

- 加速目前產品、商業或基建管線一個階段。
- 對宿敵發表進行緊急反制。
- 啟動重大發表、合規衝刺、價格戰或生態合作。
- 處理需高層拍板的事件選項。
- 啟動主要戰略轉型或財務重整。

高層指令不能用現金無限購買，確保玩家需要在進攻、反制與救火之間選擇。

### 5.4 三條勝利路線

#### 消費者霸主

1. 推出該世代消費者加權品質第一的旗艦模型。
2. 消費者市場市佔達 35%。
3. 主要宿敵發表後，仍維持市佔門檻與正常推理服務兩個 Cycle。

#### 企業信任

1. 安全品質達到當代企業門檻；Phase A 初始公式為 `max(15, 主要宿敵安全品質 × 0.9)`。
2. 企業市場市佔達 30%，且有效定價不低於企業參考價。
3. 通過連續兩個 Cycle 的 SLA／監管壓力測試。

Phase A 的 SLA 測試以安全品質、推理餘裕與營運人力計算；Phase C 導入實際合約後，以承諾履約資料擴充，勝利語意不變。

#### 開發者生態

1. 效率與速度的開發者加權品質排名第一。
2. 開發者市佔達 35%，價格不高於有效參考價的 90%。
3. 價格戰期間仍維持正現金流，且推理利用率低於 80%，連續兩個 Cycle。

#### 5.4.1 終局反攻

完成路線前兩項後，主要宿敵開始兩 Cycle 的公開終局反攻。玩家必須在該窗口守住全部條件；失敗不會結束遊戲，可重新建立條件再觸發一次，但宿敵反攻模板會換一個分支。

### 5.5 戰役結算與 Prestige

本節的「正常估值專利」沿用既有公式：`floor(sqrt(PeakValuation / PatentK))`。

勝利後可選：

1. **Prestige：**立即結算並開始新局。
2. **繼續稱霸：**進入無盡模式，挑戰其他路線目標與更高估值。
3. **稍後決定：**保留勝利狀態，下次登入再選。

第 18 Cycle 後未勝利可策略退出；取得 `floor(正常估值專利 × 0.5)`，但不取得勝利徽章與 Legacy Draft。

完整勝利帶走：

- **專利：**延續正常估值專利公式與既有數值型永久升級。
- **路線徽章：**解鎖新路線能力分支、宿敵難度或事件鏈。
- **Legacy Draft：**三選一帶入下一局。Phase A 初始池包含保留一階副戰略、取得完整宿敵傳聞、保留一個起始科技；Phase C 再加入基建藍圖與商業關係。

Legacy Draft 只作用於緊接的下一局，選定後即消耗；路線徽章與已購 Prestige 節點才是永久狀態。

不永久帶走模型、完整市佔、大量資產或整條公司能力，避免下一局失去取捨。

## 6. B — 動態事件鏈

### 6.1 事件生命週期

每條重大事件跨 2–4 Cycle：

1. **前兆：**公開觸發原因、倒數、影響範圍與風險。
2. **決策：**提供 2–3 個有明確代價的策略選項。
3. **結果：**依選項與公司可見狀態確定性計算；需要隨機時使用存檔 seed。
4. **餘波：**改變 Roadmap、合約、路線進度、後續事件或持續 modifier。

選擇與結果寫入戰役歷史，後續事件能引用，不做「選完即忘」。

### 6.2 每局事件組成

- **產業氣候 ×1：**AI 熱潮、融資寒冬、監管浪潮、能源危機等；開局揭露。
- **宿敵事件鏈 ×2–3：**旗艦發表、價格戰、開源攻勢、人才挖角。
- **公司風險鏈 ×1–2：**由低安全、過載、負債、失約等玩家狀態觸發。

每 Cycle 最多一個需要玩家處理的重大事件。小新聞只進 Board Report；同一局不重複抽取已完成的同 ID 事件鏈。

### 6.3 選項品質規則

- 不允許重大負面效果沒有前兆。
- 不允許永久固定成「花錢＝好、不花＝壞」。
- 選項應改變至少一個後續分支、Roadmap 或承諾。
- 流派、科技、明星、產品與合約狀態可解鎖或改寫選項。
- 玩家必須能在確認前看到直接成本、持續時間與已知風險。

### 6.4 首批內容

Phase B 第一版交付 8 條完整事件鏈：

- 2 條產業氣候。
- 消費者、企業、開發者各 1 條路線專屬鏈。
- 2 條宿敵鏈。
- 1 條公司風險鏈。

每條 3–4 階段、至少兩個分歧結果。現有十個一次性 industry events 可重用題材，但需轉譯成 Campaign chain；短期隨機 modifier 仍可保留為小新聞或 chain 的結果 primitive。

## 7. C — 深度營運

### 7.1 三條營運管線

#### 產品管線

研究／設計 → 訓練 → 發佈 → 成熟 → 舊版 → 下架。

- 基礎同時一個重大模型專案；科技或明星能力可擴充。
- 可由完成的母模型建立較便宜、較快的衍生模型，以部分品質交換效率或定位。
- 同區隔多模型會產品內耗，但可用價位與特性覆蓋不同需求。
- 舊模型可降價留客、遷移用戶或下架節省推理算力。
- 有未履行合約的模型不可無代價立即下架。

#### 商業管線

市場機會 → 報價 → 承諾 → 交付 → 續約／擴張。

- 同時推進一個重大商業專案。
- 企業合約：高價長約，要求安全、容量與 SLA。
- 開發者合作：增加採用與 API 用量，要求價格、相容性與吞吐。
- 消費者發表活動：短期品牌與用戶爆發，要求預留推理算力。
- 不生成數十筆小合約；每個機會都必須足以改變策略。

#### 基建管線

需求預測 → 租用保留／採購 → 建置 → 上線。

- 即時雲租用：無 lead time、單價最高、可快速縮放。
- 容量保留／長約：一個 Cycle 生效，單價較低但有最低期限。
- 自建：成本最低、有機房約束與多 Cycle lead time。
- TUI 以容量方案操作，不要求逐台伺服器維護。

### 7.2 團隊配置

保留聚合人力，不逐人排班。每 Cycle 可調整四職能投入比例：

- 研究員：模型專案／科技。
- 工程：訓練加速／基建建置。
- 營運：SLA／可靠性／事故防護。
- 行銷：品牌／商業機會。

員工 tier 必須在 TUI 可選；解雇與縮編也必須有正式入口。明星員工提供會改變規則的簽名能力，例如第二產品管線、訓練池彈性轉推理、免疫一次安全事故，而不只提供倍率。

### 7.3 CEO 戰情室

每次登入第一屏應顯示：

- 自上次登入以來的董事會摘要。
- 當前主要戰略、階段、勝利條件與缺口。
- 主要宿敵兩步 Roadmap。
- 現金跑道與每 Cycle 損益。
- 下一次產品／活動後的預估推理缺口。
- 最接近違約的承諾。
- 三條管線的狀態與完成 Cycle。
- 當前重大 Pending Decision 與本 Cycle 高層指令是否已使用。

市場、模型、算力、團隊、科技頁維持各自的詳細操作；戰情室只做跨系統判讀與導流。

## 8. 技術架構與資料流

### 8.1 模組邊界

#### `internal/model/campaign.go`

概念型別：

- `CampaignState`
- `DoctrineState`
- `RivalRoadmap`
- `RouteProgress`
- `CampaignPipeline`
- `ActiveEventChain`
- `PendingDecision`
- `BoardReport`

#### `internal/balance/campaign_*.go`

拆分公司流派、勝利條件、對手 Roadmap、事件鏈、合約與基建 catalog，避免繼續擴大現有 `balance.go`。

#### `internal/sim/campaign_*.go`

- 純函式 Cycle 推進。
- 路線進度與勝利判定。
- 對手行動。
- 事件鏈轉移。
- 三條管線結算。

#### `internal/sim/apply.go`

新增經驗證的玩家命令：選擇／轉換戰略、發出高層指令、處理事件、接受／違約、啟動下架／採購／建置、結算戰役。

#### `internal/tui`

- 新增戰略頁或將總覽升級為 CEO 戰情室。
- 預測與可執行性一律呼叫 `sim` view helpers，不在 TUI 複製公式。
- 重大命令不可使用靜默失敗的 `applyOK()` 路徑。

### 8.2 純函式邊界

- TUI／啟動層讀 wall-clock 並計算到期 Cycle 數。
- `AdvanceCampaignCycle(state, config)` 本身不讀時間、不 I/O、不使用全域亂數。
- 隨機選擇只讀寫 `CampaignState` 的確定性 RNG state。
- Catalog 物件在存檔只保存穩定 ID，不序列化整份設定。
- Board Report 是每次 Cycle 結算結果，保存最近 20 份，超過丟最舊。

### 8.3 Typed effects

流派、Roadmap、合約與事件鏈使用有限的效果 primitive：品質／科技成本、市場參考價、用戶成長、信任／採用、現金／R&D、算力、管線時間、Roadmap 延遲，以及持續若干 Cycle 的 modifier。

特殊內容可有小型專用 handler，但設定檔不嵌入任意腳本。

## 9. 錯誤、失敗與公平保護

### 9.1 命令錯誤

戰略命令不可靜默 no-op。TUI 必須持續顯示：

- 缺少的資源數量。
- 被哪條管線或承諾阻擋。
- 合約未滿足的安全、容量或人力門檻。
- 轉型、結算或事件選項不可用的具體原因。

### 9.2 預覽與二次確認

轉型、接受高風險合約、提前違約、下架有用戶／合約的模型、Prestige、策略退出、大額自建都需要預覽與確認。預覽至少包含成本、完成 Cycle、現金跑道、容量缺口、路線進度與違約風險。

### 9.3 財務危機

取消目前超過負債門檻立即重開的戰役行為：

1. 現金跌破零時進入財務危機。
2. 玩家有兩個董事會 Cycle 重整。
3. 可用緊急融資、縮編、減租、出售資產、提前終止產品／合約止血。
4. 緊急融資會降低本局估值倍率或路線進度，不能無成本續命。
5. 兩 Cycle 後仍無法恢復才強制策略退出；取得 `floor(正常估值專利 × 0.5)`，不給徽章與 Legacy Draft。

### 9.4 防止無解雪球

- 重大對手行動提前至少一 Cycle 公開。
- 失去市佔後可出現一次反攻任務，成功取得短期搶先窗口。
- 對手可奪走優勢，但不可永久封鎖勝利條件。
- 終局反攻失敗可以重新挑戰。
- 第 18 Cycle 後允許策略退出。

### 9.5 存檔版本漂移

- 新 Campaign 欄位零值可讀。
- 舊存檔首次開啟不重置模型、現金或科技；依現況提出流派建議，由玩家確認。
- 缺失 catalog ID 不 panic；取消尚未執行的 pending 項目並釋放管線。
- 已扣款但無法完成的重大專案依 catalog migration policy 退款，並在 Board Report 留紀錄。

## 10. 測試策略

### 10.1 單元測試

- 相同 state、seed、命令與 config 產生相同 Cycle 結果。
- Cycle 固定處理順序。
- 流派選擇、轉型限制、成本與進度損失。
- 對手 Roadmap 不讀尚未提交的玩家命令。
- 勝利必須守住兩 Cycle，不能只瞬間碰門檻。
- 事件鏈每個分支、餘波與持續效果。
- 三條管線完成、取消、違約與 slot 釋放。

### 10.2 Property／Invariant 測試

- 最多一個重大 Pending Decision。
- 離線不自動新增支出。
- 專案、合約、事件與 Prestige 不重複扣款或發獎。
- 員工、伺服器與管線計數不出現無效負值。
- 事件鏈不進入不存在的 stage。
- 三條勝利路線都有可達狀態。
- 舊存檔缺 Campaign 欄位仍可載入。

### 10.3 18 Cycle 情境測試

每條路線至少覆蓋：正常勝利、中途轉型、被壓制後反攻、財務重整成功、失敗退出、離線 8／24／72 小時、勝利後 Prestige、勝利後繼續無盡。

### 10.4 TUI 測試

- Board Report 能說明發生什麼、原因與下一步。
- 對手兩步情報與倒數正確。
- 不可執行選項顯示原因。
- 重大操作預覽包含必要風險。
- 錯誤不會被下一個 Tick 清除。
- 離線摘要區分資源、對手、事件與待決策。

### 10.5 Balance 模擬

使用低、一般、高三種 token 輪廓，搭配三路線與多個宿敵 seed 批次模擬：

- 熟悉玩家約 9–21 Cycle 首勝，對應 3–7 個真實日。
- 每次登入有 2–3 個有意義決策，最多一個重大 pending。
- 不存在固定唯一最佳開局或路線。
- 高 token 能加速，但不能跳過產品、算力與市場考驗。
- 低 token 仍可推進，不會完全卡死。

預設不傳遠端 telemetry；提供本機戰役報告匯出供 playtest 比較。

## 11. 分階段交付

### Phase A：戰略戰役骨架

1. Campaign clock、Board Report、離線最多三 Cycle。
2. 三種主要戰略、路線進度、能力選擇與轉型。
3. 主要宿敵／攪局者 Roadmap 與兩步情報。
4. 高層指令與三條勝利路線。
5. 戰役結算、路線徽章與初始 Legacy Draft。

### Phase B：動態事件鏈

1. 通用事件鏈狀態機與 typed effects。
2. 前兆、決策、結果、餘波介面。
3. 首批 8 條事件鏈。
4. 與 Roadmap、流派與勝利進度串接。

### Phase C：深度營運

1. 模型生命週期、衍生、內耗與下架。
2. 商業機會、合約、SLA 與發表活動。
3. 容量保留、自建 lead time 與預測。
4. 團隊投入比例與明星簽名能力。
5. 財務危機與完整重整流程。

Phase A、B、C 各自需要獨立 implementation plan、驗收測試與 playtest gate。Phase B 不阻擋 A 發布；Phase C 不阻擋 B 發布。

## 12. 成功標準

1. 玩家在每次 5–10 分鐘登入內，看得懂局勢並做出 2–3 個有意義決策。
2. 對手行動至少提前一 Cycle 可見，玩家能說明失敗原因與可用反制。
3. 三條路線的資源配置、風險與勝利條件明顯不同。
4. 一局在正常 token 使用下約 3–7 天完成，且不是靠純粹拉高成本。
5. 勝利後有 Prestige 與繼續無盡兩種合理選擇。
6. 重玩差異來自流派能力、宿敵組合、事件鏈與 Legacy Draft，而非只把相同流程跑快。
7. 離線一天不會因未處理的大量事件形成無解局面。
8. 深度營運增加策略承諾，但不要求逐員工、逐伺服器或大量小合約管理。
9. `sim` 仍保持純函式、確定性與可重放。
10. 每一 Phase 均可獨立測試、平衡與發布。
