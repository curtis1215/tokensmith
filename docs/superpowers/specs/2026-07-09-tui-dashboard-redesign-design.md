# TUI Dashboard Redesign — Design

- **日期**：2026-07-09
- **狀態**：設計已確認，待寫實作計畫
- **範圍**：六頁 TUI 儀表板重做（Lipgloss 多欄卡片）+ 共用殼層；不改經濟公式、不做真事件系統
- **相關**：`docs/superpowers/specs/2026-07-07-tokensmith-design.md` §11；現況 `internal/tui/*`

---

## 1. 問題

目前 TUI **功能完整但呈現像 debug 列表**：

- 資訊層級弱（少卡片、少 bar、科技顯示 raw node ID）
- 決策因果不明（改價 / 租推理後，用戶與負載關係看不見）
- 操作語言不一致（有的頁有 cursor，有的只有快捷鍵）
- 與設計 mockup（§11.3）落差大：缺市佔條、總覽 2×2、統一 footer

玩法平衡在 v0.5.3 已可用；本輪專注 **可讀性與儀表板感**。

## 2. 目標與非目標

### 目標

1. **六頁**（總覽 / 模型 / 市場 / 算力 / 團隊 / 科技）全部改為 Lipgloss **多欄儀表板**。
2. **共用殼層**：標題 + Day、資源條（可染色）、Tab、注意列、統一 footer help。
3. **鍵位與 `Apply` 命令語意不變**（`t/p/$/r/i/h/e/o/k/s/Enter…`）。
4. 必要時加 **純函式 view helper**（市佔 bar、威脅、可撐用戶、科技中文名）；**不改** tick / 經濟公式。
5. 事件列用 **注意 / notice** 占位，不實作產業事件池。

### 非目標

- 真事件系統與決策選項 UI
- 自幹 responsive grid 引擎
- streak 真實計數（無資料則省略或「—」）
- 重做訓練/發佈 dialog 交互（僅輕微框線/排版對齊）
- 估值 / 用戶 / 對手 catchup 再平衡

### 風格決策（已確認）

| 項 | 選擇 |
|---|---|
| 範圍 | 六頁一起重做 |
| 版面 | Lipgloss 多欄儀表板（卡片 + bar + JoinHorizontal） |
| 事件 | 不做真事件；pressures + notice |
| 落地法 | **方案 A**：layout kit → 殼 → 六頁依序換皮 |

## 3. 殼層與視覺語言

### 3.1 目標終端

- 寬 **≥ 80** 欄（理想 100–120）。過窄時卡片改 **上下堆疊**（同一套 vstack，不做複雜 grid）。
- 總覽主區盡量一屏；科技/對手列表可變長。

### 3.2 骨架

```
┌─ Tokensmith ──────────────────────── Day N ─┐
│ 💰 $…  ⚡R&D … (+…/s)  🖥訓..% 推..%  📈$…   │
├──────────────────────────────────────────────┤
│ [1]總覽  [2]模型  …  [6]科技                  │
├──────────────────────────────────────────────┤
│              （當頁內容）                      │
├──────────────────────────────────────────────┤
│ ⚠ 注意…（0–3 則；無則省略整列）                 │
│ [本頁快捷鍵…]              [Tab]切頁 [q]離開   │
└──────────────────────────────────────────────┘
```

- Dialog（訓練 / 發佈）蓋住 **page 區**；資源條保留。
- `m.notice` 放在 **標題下、資源條上**；單行；本輪不做多則 queue。

### 3.3 Layout kit

新建例如 `internal/tui/layout.go` + `style.go`（可合併一檔若更簡單）：

| API（示意） | 職責 |
|---|---|
| `card(title, body string) string` | rounded border + 標題 |
| `hrow(parts ...string) string` | 橫向拼卡 |
| `vstack(parts ...string) string` | 直向 |
| `kv(label, value string) string` | 鍵值列 |
| `bar(frac float64, width int) string` | 進度條（沿用/包一層 progressBar） |
| `footer(pageKeys string) string` | 本頁鍵 + 全域 Tab/q |
| `truncate(s string, maxRunes int) string` | 中文安全截斷 |
| styles: `muted`, `accent`, `warn`, `ok`, `title`, `tabActive` | Lipgloss |

### 3.4 色碼

| Token | 用途 |
|---|---|
| accent（既有 205） | 選中 Tab、▸ cursor、標題強調 |
| warn | 推理 ≥90%、待發佈、無模型、現金 < 0 |
| ok / muted | 已解鎖、次要數字 |

資源條：推理 util ≥ 0.9 時「推理%」用 warn；現金 < 0 時現金 warn。

## 4. 六頁資訊架構

### 4.1 總覽

```
[公司 KPI]     [訓練 / 發佈狀態]
[市佔·消費者]  [算力壓力 + 里程碑]
[注意 全寬]
```

| 卡 | 內容 |
|---|---|
| 公司 | 估值、總用戶、消費者排名、月營收 |
| 訓練/發佈 | 訓練中：Gen、bar、區隔；否則無訓練；若有草稿：「待發佈 N · 模型頁 p」 |
| 市佔 | 你 + 主要對手相對 bar（消費者區隔；資料用現有 appeal/share 推導） |
| 算力+里程碑 | 訓/推 util bar；下一估值里程碑 bar |
| 注意 | `pressures()` |

快捷：`[t]訓練` 等維持（總覽可開訓）。

### 4.2 模型

- **列表**：待發佈 / 營運中分區；▸ 游標；一行摘要（名、Gen、區隔、用戶、$）。
- **詳情卡**（選中）：
  - 四維 quality bar + 數值
  - 上線：用戶 → `EstimateUserTarget` 封頂；粗月收；該模對推理 load 貢獻；全公司推池 util
  - 定價 vs `EffectiveRefPrice`
  - 草稿：強調待發佈 + `p`
- 窄屏：列表上、詳情下。
- 鍵：`↑↓ t p $` 不變。

### 4.3 市場

- 三區隔各一卡：你的用戶、排名、你 vs 對手 top 的 bar。
- 對手檔案卡：名、能力 bar、專長維度、**威脅 高/中/低**（相對玩家該區隔最佳模型 appeal）。
- 本輪無對手長文 lore。

### 4.4 算力

- **因果卡**（新）：
  - 訓/推池 bar + 有效算力
  - `可撐用戶 ≈ EffectiveInference / InferenceLoadPerUser`（容量 0 → 文案「未配置推理 · grace 不砍用戶」）
  - 現況總用戶；可選主模型/總市場目標粗值
  - 缺口提示（需加約 n 張入門推理）
- **製程表** + 機房電力/空間 bar（保留現鍵 `r/R i/I b/B e ↑↓`）。

### 4.5 團隊

- 四職能卡：人數、效果摘要、對應快捷鍵。
- 可選：薪資合計 /s（純展示）。
- 明星卡：姓名、已簽/未簽、**加成一句**（從 StarEffects 摘要）、簽約成本。
- `[h/e/o/k/s]` 語意不變；`s` 仍可簽「第一位未簽」（本輪可不做明星 cursor，除非實作順手）。

### 4.6 科技

- 按分支分組標題（演算法 / 硬體 / 商業 / 對齊）。
- 每節點：**中文名**、成本或 ✓/🔒、**效果一句**。
- 中文名與效果來自 tui 側 `techMeta` map（key = node ID）；不改 balance ID。
- `↑↓ Enter` 解鎖不變。

### 4.7 Dialog

- 訓練 / 發佈：沿用邏輯；外觀改用 `card` 風格與 footer 一致即可。
- 不改命名/定價/預估公式。

## 5. View helpers（sim 或 tui）

優先放 `internal/sim` 若屬遊戲事實；純展示字串可放 tui。

| Helper | 用途 |
|---|---|
| 市佔/appeal 序列（按區隔） | 總覽與市場 bar |
| 威脅等級 | 市場對手 |
| 可撐用戶 | 算力因果卡 |
| `techMeta` | 科技中文名 + 效果 |
| 明星效果一句 | 團隊 |

全部 **pure**、可單測。

---

## 6. 實作切分（建議 commit / PR 順序）

單一 feature branch 可多 commit：

1. **Layout kit + 殼層**（View 骨架、footer、色碼、notice 位置）— 舊 page 先塞進新殼，行為不變。
2. **總覽** 儀表板。
3. **模型** 列表+詳情。
4. **市場** 三區隔 + 對手。
5. **算力** 因果卡 + 表重排。
6. **團隊** 職能/明星卡片。
7. **科技** 中文 meta + 分組。
8. **回歸**：全鍵測、snapshot 關鍵字、`go test ./...`。

可一個 PR 或拆「殼+總覽+模型」/「其餘四頁」兩 PR；預設 **一個 PR** 若體量可控。

## 7. 測試策略

| 層 | 內容 |
|---|---|
| Layout | card/hrow 非空、truncate rune 邊界 |
| 各頁 | 既有 `strings.Contains` 測更新期望字串；保留鍵位測（`t` 開訓、`p` 發佈等） |
| Helper | 市佔排序、可撐用戶、techMeta 齊全 |
| 手動 | 80 欄與 120 欄各看一輪；發佈後總覽/模型/算力三頁聯動 |

不要求像素級 visual regression。

## 8. 相容與風險

| 風險 | 緩解 |
|---|---|
| 窄終端爆版 | 過窄改 vstack；手動 80 欄檢查 |
| 中文對齊 | truncate + lipgloss；避免手算 byte 寬 |
| 測試脆 | 斷言穩定關鍵詞（「待發佈」「可撐用戶」），少鎖整屏 |
| 範圍膨脹 | 事件/streak/改鍵位明確 out of scope |

舊存檔無 UI 狀態；無 migration。

---

## 9. 成功標準

1. 六頁均為卡片/多欄儀表板，不再是純純文字列表。
2. 總覽 10 秒內能回答：有沒有模型、訓完了沒、推池危不危險、下一步按哪。
3. 模型頁能回答：封頂多少、定價偏高偏低、推池夠不夠。
4. 算力頁能回答：現在能撐多少用戶。
5. 科技頁不暴露裸 `algo-cap-1` 作為唯一可讀名。
6. `go test ./...` 全綠；主要快捷鍵行為不變。
